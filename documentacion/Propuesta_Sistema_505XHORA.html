Propuesta Técnica: Sistema de Gestión Desktop

Microempresa 505 X HORA — Arquitectura Offline-First para Operaciones Comerciales
Resumen Ejecutivo: Sistema desktop offline-first diseñado específicamente para el contexto operativo, legal y tecnológico de una microempresa en Cuba (TCP, ONAT, conectividad limitada). Prioriza la trazabilidad del flujo de ventas, el cuadre final diario y la legalización contable.

1. Stack Tecnológico Recomendado

Capa Tecnología Justificación
Frontend/UI Tauri (Rust) + React o Electron + React Interfaz moderna y profesional. Tauri consume mínima RAM (crítico en equipos modestos).
Base de Datos SQLite local Funciona 100% offline. No requiere servidor. Respaldo copiando un solo archivo.
Reportes/Impresión PDF nativo + impresora térmica/matricial Para facturas, vales de venta/entrega y reportes ONAT.
Seguridad Usuarios locales con roles (RBAC) Cada perfil visualiza únicamente sus funciones autorizadas.
Alternativa conservadora: Si los equipos son muy limitados (Windows 7/8, 2-4GB RAM), utilizar C# WinForms o WPF con SQLite. Es visualmente más tradicional pero infinitamente más rápido en hardware bajo.

2. Arquitectura por Módulos (Roles)

La aplicación es monolítica modular. Al iniciar sesión, el sistema carga exclusivamente las funciones del rol asignado:

2.1 Módulo Comercial (Oficina)

  • Vale de Venta: Generación con número correlativo, datos del cliente, productos y condiciones.
  • Facturación: Emisión del documento legal en formato para impresora.
  • Liquidación Rápida: Registro inmediato del efectivo/transferencia entregado por el mensajero.
  • Cuadre Parcial: Vista en tiempo real de ventas del día vs. fondos recibidos.

2.2 Módulo Almacén (Ayudante)

  • Vale de Entrada: Recepción digital del vale desde el módulo comercial.
  • Checklist de Calidad:
    • Verificación de envoltura y embalaje
    • Funcionamiento de equipos
    • Fecha de caducidad de perecederos
  • Conteo Físico Diario: Interfaz optimizada para conteo de existencias al cierre.
  • Alertas de Stock: Notificación de productos bajo nivel mínimo.

2.3 Módulo Mensajero (Tablet/App ligera)

  • Conciliación Previa: Registro del acuerdo de pago y plazo antes de la salida.
  • Estado de Entrega: "En tránsito", "Entregado", "Incidente".
  • Cobro: Tipo (efectivo/transferencia), monto, foto del comprobante.
  • Notificación de Incidentes: Registro textual de problemas con el cliente.

Nota: Si no hay presupuesto para desarrollo móvil, el mensajero utiliza una tablet con la misma aplicación desktop en modo "Mensajero".

2.4 Módulo Custodio

  • Recibo de Custodia: Pantalla simplificada: "Recibo X pesos, Y productos". Confirmación mediante PIN.
  • Historial de Custodia: Trazabilidad completa de entregas con fecha, hora y responsables.

2.5 Módulo Administrador / Propietario

  • Dashboard: Ventas del día, semana y mes con comparativas visuales.
  • Gestión de Gastos: Alquiler, ONAT, combustible, mantenimiento, salarios.
  • Nómina Quincenal:
    • Salario base configurable por trabajador
    • Comisiones automáticas por volumen de ventas
    • Condicionado al comportamiento de ventas (política empresarial)
  • Configuración General:
    • Horarios: Atención al cliente (09:00-16:00), Personal (08:30-17:00)
    • Mantenimiento de transporte: Alertas los sábados
    • Eliminación del rubro alimentación (según política de optimización de costos)

3. Flujo de Datos Implementado

[Cliente llega]
    ↓
[Comercial] → Genera Vale de Venta + Factura
    ↓
[Almacén] ← Recibe Vale de Entrega digital
    ↓
[Checklist físico] → Marca verificaciones → Entrega a Mensajero
    ↓
[Mensajero] ← Concilia pago/plazo → Traslado → Cobro → Regresa
    ↓
[Comercial] ← Liquidación inmediata de fondos
    ↓
[Almacén] ← Conteo físico del día
    ↓
[Custodio] ← Cuadre final (ventas vs. existencias vs. efectivo)
    ↓
[Administrador] ← Revisa gastos, nómina, reportes ONAT

4. Plan de Implementación MVP

Para no demorar la puesta en marcha, se propone el siguiente orden de prioridad:

Fase 1 (Semanas 1-4): Operación Básica
  1. Catálogo de productos (nombre, categoría, precio, stock).
  2. Usuarios y roles (Comercial, Almacén, Admin).
  3. Vale de Venta + Factura (impresión).
  4. Liquidación del mensajero (registro de cobros).
  5. Cuadre final del día (cierre de caja).
Fase 2 (Semanas 5-8): Control y Legalidad
  1. Conteo físico de almacén con ajustes de inventario.
  2. Módulo de gastos operativos (ONAT, alquiler, combustible).
  3. Reportes para ONAT (ingresos y gastos declarables).
  4. Custodia y traspaso nocturno.
Fase 3 (Semanas 9-12): Optimización
  1. Nómina quincenal automática con comisiones.
  2. Alertas de mantenimiento (sábados).
  3. Métricas de disciplina de venta (checklist de apariencia/vocabulario).
  4. Respaldo automático de la base de datos (USB o carpeta local).

5. Consideraciones Específicas para el Contexto Cubano

Reto Solución en el Sistema
Internet intermitente Todo funciona offline. Respaldo en la nube es opcional y no requerido.
ONAT / TCP Reporte pre-formateado de ingresos y gastos mensuales, listo para declaración.
Equipos limitados SQLite no consume recursos. Interfaz ligera sin efectos pesados.
Impresoras Soporte para impresoras térmicas baratas (58mm/80mm) para vales y facturas.
Legalización futura El sistema ya estructura roles, flujos y finanzas como exige una entidad formal.

6. Protocolo de Cuadre Final y Seguridad

Pantalla obligatoria antes de las 17:00:

  1. Cuadre de Caja: El sistema calcula automáticamente: Ventas facturadas - Gastos del día = Efectivo esperado.
  2. Conciliación: El comercial ingresa el efectivo físico contado.
  3. Diferencia: Si existe faltante o sobrante, el sistema exige justificación obligatoria antes de continuar.
  4. Custodia: Solo cuando el cuadre coincide (o se justifica la diferencia), se genera el acta de entrega al custodio con fecha, hora y confirmación PIN de ambas partes.

7. Recomendación Concreta

Empieza con Tauri + React + SQLite si cuentas con alguien con conocimientos web, o C# + SQLite si tienes un desarrollador de Windows tradicional.

Primer entregable crítico: Un sistema donde el comercial cree un vale de venta, el almacén marque que revisó el producto, el mensajero registre el cobro al volver, y el admin vea si cuadra la caja al final del día.

Esto resuelve el 80% del dolor operativo inmediato. Todo lo demás (nómina, ONAT, mantenimiento) se construye sobre esa base sólida.

8. Escalabilidad Futura

  • Ahora: App desktop local con SQLite.
  • Después: Migrar la base de datos a un servidor local (Raspberry Pi o PC vieja) para acceso multi-puesto por red local.
  • Futuro: Sincronización con API ligera si se requiere conectar con proveedores o socios estratégicos.
